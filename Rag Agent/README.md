# Company Knowledge Base — RAG Agent

Retrieval-augmented chat agent for the company's FAQ / services / pricing
docs, with a web chat UI and per-session conversation history. Embeddings
are local and free (no API key); the agent's answers are generated by
Google Gemini (needs a free API key).

## Setup

```bash
pip install -r requirements.txt
python ingest.py
```

`ingest.py`'s first run downloads the `BAAI/bge-small-en-v1.5` embedding
model (~130MB, runs on CPU, no API key, completely free). This only happens
once — it's cached locally afterward. It creates `chroma_db/` in this
folder — a persisted, on-disk vector store. Delete that folder and re-run
`ingest.py` any time you edit the source `.md` files in `data/`.

Then set up your Gemini key:

```bash
cp .env.example .env
# edit .env and paste your key from https://aistudio.google.com/apikey
```

## Run the web chat UI

```bash
python server.py
```

Open http://localhost:8000 — enter a question and get an answer grounded in
the knowledge base. Each browser tab gets its own session (a random
`session_id` stored in `localStorage`), so conversation history stays
independent per visitor and follow-up questions ("what about tier 3?")
resolve correctly. Use "New chat" to reset a session.

## Run the agent from the CLI

```bash
python agent.py                                          # interactive loop
python agent.py "does tier 2 include 3D renderings?"      # one-shot
```

## Test retrieval only (no LLM/API key needed)

```bash
python query.py "does tier 2 include 3D renderings?"
python query.py "what happens if you find problems after opening the walls?"
python query.py "how many rounds of revisions do I get?"
```

## Files

- `chunker.py` — custom markdown-aware chunker. Splits each doc by its
  natural structure (FAQ = one chunk per Q&A, Services = one chunk per tier
  + a separate comparison-table chunk, Company Info = one chunk per section).
  Every chunk gets a contextual prefix (`Document: ... / Section: ...`)
  added *only* for embedding — the LLM still sees clean chunk text at
  answer time.
- `ingest.py` — embeds all chunks and writes them into `chroma_db/`.
- `query.py` — quick CLI to sanity-check retrieval quality (no LLM).
- `agent.py` — the actual agent: wraps the retriever as a
  `search_knowledge_base` tool for a Gemini-powered LlamaIndex
  `FunctionAgent`, and manages one conversation history per `session_id`
  (`RagAgent.chat(session_id, message)`). Also runnable as a CLI chat loop.
- `server.py` + `static/index.html` — FastAPI backend and chat web page.
  `/api/chat` takes `{message, session_id}` and returns `{reply,
  session_id}`; `/api/reset` clears a session's history.

## Notes / possible improvements

- **Sessions are in-memory** — they reset if the server restarts, and won't
  work if you ever run multiple server processes/replicas behind a load
  balancer (each would have its own memory). Fine for a single-process
  deployment; if you outgrow that, swap `RagAgent`'s `Context` dict for a
  Redis- or SQLite-backed store.
- **Embeddings are local/free** and already tuned to this doc set via the
  custom chunker — no changes needed there unless you add new source docs
  (in which case just add a chunker function in `chunker.py` and re-run
  `ingest.py`).
- **Booking a meeting** isn't wired up yet. To add it, give the agent a
  second tool, `book_meeting(name, email, datetime) -> str`, backed by the
  Google Calendar API (`google-api-python-client` +
  `google-auth-oauthlib`), and tell it (in `SYSTEM_PROMPT`) to only call it
  once it has collected name, email, and a specific date/time — not
  speculatively. Needs a Google Cloud project with the Calendar API enabled
  and either an OAuth flow or a service account + your business calendar ID.
